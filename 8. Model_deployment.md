# MLOps Deployment Notes — Deploying ML Project to AWS Elastic Beanstalk

## 1. Why Deployment is Important

After completing an **end-to-end ML project**, the next important step is **deployment**.

Deployment means:

```text
Making your ML application available on the internet so users can access it.
```

For example:

User → Web Application → ML Model → Prediction

If deployment is not done, your ML project remains only on your **local machine**, which is not useful in real-world production.

# 3. AWS Elastic Beanstalk

### What is Elastic Beanstalk?

AWS Elastic Beanstalk is a **cloud service that allows you to deploy and run web applications easily**.

Think of it as:

```text
Elastic Beanstalk = Preconfigured Cloud Server
```

Technically:

```text
Elastic Beanstalk provides a Linux machine (instance)
where your application runs.
```

It automatically handles:

* Server provisioning
* Scaling
* Deployment
* Monitoring

# 4. Two Important Configuration Files Required

Before deployment, we must create **two important configurations**.

### Configuration Folder

Create a folder:

```
.ebextensions
```

Inside it create:

```
python.config
```

---

## Purpose of `.ebextensions/python.config`

This file tells AWS:

```text
What is the entry point of your application?
```

Entry point = The file where the application starts running.

---

# 5. Configuration File (`python.config`)

## Full Code

```yaml
option_settings:
  aws:elasticbeanstalk:container:python:
    WSGIPath: application:application
```

---

## Explanation

### option_settings

This defines configuration settings for Elastic Beanstalk.

---

### aws:elasticbeanstalk:container:python

This tells AWS that the application uses:

```text
Python container
```

If we were using **Docker**, the configuration would be different.

---

### WSGIPath

```text
WSGIPath: application:application
```

This means:

```
file_name : variable_name
```

So here:

```
application.py → application
```

Meaning:

```python
application = Flask(__name__)
```

---

# 6. Creating `application.py`

In our project we originally had:

```
app.py
```

But Elastic Beanstalk expects:

```
application.py
```

Therefore we create a new file:

```
application.py
```

and copy the entire code from `app.py`.

---

## Important Change

Remove debug mode.

### Original

```python
app.run(host="0.0.0.0", debug=True)
```

### Updated

```python
app.run(host="0.0.0.0")
```

Reason:

```text
Debug mode should never be enabled in production.
```

# 8. Deployment Architecture

The deployment system will look like this:

```
GitHub Repository
       │
       │
       ▼
AWS CodePipeline
       │
       │
       ▼
Elastic Beanstalk
       │
       ▼
Running Web Application
```

Explanation:

1. Code is stored in **GitHub**
2. AWS **CodePipeline monitors GitHub**
3. When code changes → deployment triggers
4. Code gets deployed to **Elastic Beanstalk**

---

# 9. Continuous Delivery Pipeline

The pipeline used here is called:

```
Continuous Delivery (CD) Pipeline
```

Meaning:

```
Whenever code changes → it can be deployed automatically
```

Note:

CI/CD consists of two parts:

| Term | Meaning                |
| ---- | ---------------------- |
| CI   | Continuous Integration |
| CD   | Continuous Delivery    |

In this lecture we only implement **CD pipeline**.

# 13. Fixing Deployment Error

Initially deployment failed.

Reason:

There were two files:

```
app.py
application.py
```

Elastic Beanstalk expects only:

```
application.py
```

Solution:

Delete:

```
app.py
```

Keep only:

```
application.py
```

After fixing this, deployment succeeded.

# 16. Automatic Deployment from GitHub

If you push new code to GitHub:

Example change:

```
requirements.txt
```

Even adding a blank line will trigger detection.

AWS CodePipeline detects the change.

It asks:

```
Release change?
```

If you confirm:

```
New deployment starts automatically
```

Pipeline runs again and updates the application.

11. Select Python platform
12. Create environment
13. Create AWS CodePipeline
14. Connect GitHub repository
15. Skip build stage
16. Configure deploy stage
17. Deploy to Elastic Beanstalk
18. Fix errors if deployment fails
19. Access application URL
20. Test predictions
21. Push code updates to trigger redeployment
